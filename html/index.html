Introduction

The gradido Node is inspired by bitcoin.
The target is getting a node powerful enough to handle in future every transaction between every human on earth.
That means approx. 14 transaction... per ms.
Currently approx. 3.5 billion transactions per day (digital and analog together) exist.

The world population is divided in regional groups approx. 100 per group.
Groups can have other groups as parent, recursive.
The Node Server has a separate block chain for every group he is listening.
14 transactions per ms is only valid for a Node Server listening to all groups worldwide.
Most should listening only at some groups but should also operate resource efficiency to run on small and cheap servers
and nevertheless withstanding hacking-attacks.

Transaction Protocol

For Gradido transactions google protobuf messages are used, inspired by hedera hashgraph message format.
It all starting with a transaction. This will later be exchanged with hedera consensus message.

message Transaction {
 uint64 id = 1; // unique per group, starting with 1, counting upwards
 TimestampSeconds received = 2; // world time, received at ordering service, currently community server, later hedera hashgraph
 SignatureMap sigMap = 3; // containing signature - pubkey pairs Byte
 bytes txHash = 4; // hash for complete transaction
 bytes bodyBytes = 5; // containing TransactionBody but serialized, like hedera do it
}


message TransactionBody {
 string memo = 1; // max 150 chars, mainly encrypted with shared keys
 TimestampSeconds created = 2; // world time, creation date of transaction
// specific transactions, not yet complete.
 oneof data {
  StateCreateGroup createGroup = 6;// the first transaction for a new group
  StateGroupChangeParent groupChangeParent = 7;// change group parent
  Transfer transfer = 8;// transaction type for sending gradidos from one to another
  TransactionCreation creation = 9;// transation type for the gradido creation
 }
}
Mainly it exist two transaction types: State Changes and Gradido Transaction.

State Changes:

  • creating or moving a group
  • changing group council member
  • creating new gradido account
  • moving gradido account to another group
  • adding hash of email or nickname to gradido account
  • removing hash of email or nickname to gradido account


Gradido Transactions:

  • Transfer gradidos from one to another
  • Create gradidos through group founder
  • Create gradidos through group council members

File Formats

root folder is ~/.gradido like with bitcoin or other crypto currencies
first file is group.index
txt format, created by user
containing in this format:
<group public key> = <folder name>
The Gradido Node create the folder with this name and store every transaction belonging to this group in there
in the group folder:
 .state: leveldb format, containing:

  • current group states
  • counter for account index (lastKtoIndex)
  • counter for last block id (lastBlockNr)


 pubkeys_<first hash byte hex>
  _<second hash byte hex>.index: binary, unordered, containing:

  • account hash, kto index uint32
  • file hash at end of file

  file is unordered, sorting in memory

 blk0000000X.index, binary, sorted by kto index, containing:

  • lowest kto index, highest kto index
  • kto index, count, [count X file positions where transaction which contain kto starts]
  • file hash at end of file

 write out after block finished, until then exist only in memory

 blk0000000X.data, binary, sorted by transaction id, containing:

  • size, serialized transaction (protobuf format)
  • file hash at end of file
